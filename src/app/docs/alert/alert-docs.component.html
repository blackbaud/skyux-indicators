Alerts highlight critical information that users absolutely must see when they visit a page. {{ alertMessage }}

Use alerts when something on the page the user is viewing is important and out of the ordinary, and the user needs to know about it before digesting the rest of the page.

Use alerts when the system is in a state that suggests or requires a call to user action.

Don’t use alerts to convey a normal lifecycle status of a record or page. Use labels or status indicators instead.

Don’t use alerts to show messages about the success of failure of user actions. Use toasts instead.

Alert icon Text Close button

Danger alert – for errors or to let the user know the system is in an unusable state.

Info alert – for messages that provide helpful information or context but do not require action.

Success alert – for messages conveying a positive or successful state.

Warning alert – for messages the user should know about and perhaps act on, but that do not constitute an error state or indicate that something is wrong.

If the user should be able to dismiss the alert from the page after reading, you can include an optional close button.

Do not make an alert dismissible if it is important for users see every time or if it is triggered by underlying data.

Text inside an alert can contain hyperlinks. For accessibility reasons, these links will not be styled blue as other links in the system are.

When present, an alert should always be placed at the top of the page, above the page header.

Be careful of situations where multiple alerts may stack up; they quickly lose their value and become cluttered if several are present. Think about alternate methods of presenting the information or consolidating messages into one alert.
